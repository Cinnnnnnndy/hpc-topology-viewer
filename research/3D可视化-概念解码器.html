为什么这么表达 · 通道分工

◆编码信条：每个视觉通道只归一件事，且能一眼看见

"概念清晰"的另一半是看图不费劲。data.ts 写死了一套铁律：谁用位置、谁用颜色、谁用形状，一一分工。下面每格直接画出这个通道长什么样——记住这套编码，任何一张图你都能一眼拆开。

空间位置 / 嵌套

→ 结构（物理容纳）

"谁装在谁里面"本身就是空间关系，且静态不变——位置留给它最省脑。

红 / 黄 / 绿 + 灰

→ 运行状态（唯一一套）

铁律：一个状态 = 一个固定色、无渐变。状态最需一眼抓到，独占最扎眼的红黄绿。

中性蓝灰

→ 结构 / 类型

层级、器件类型退成没有色相的蓝灰，主动让出红黄绿，不跟状态抢眼睛。

dierank
青 teal / 靛 indigo

→ 硬件域 / 软件（rank）

硬件用青、软件 rank 用靛，刻意两种色相，让硬件和软件永远不糊在一起。

平面签名色

→ 三通信平面

一根"全互联"逻辑线其实骑三个真实物理平面（UB 绿 / RDMA 橙 / VPC 紫），各一色才能分开。

并行签名色（默认关）

→ 并行维（opt-in）

TP/PP/DP/EP 签名色和状态色会撞（PP 绿 ≈ 状态绿），默认不显、只在"通信域"镜头里开，PP 改青避让。

时间轴 + 游标

→ 时间动态

计算是动态的，但动态只改颜色、绝不动位置——否则位置通道被两件事抢，图就读不懂了。游标扫过 → 卡的颜色变、位置不动。

方案里三个最容易混的点

⊞"整网"是两件事 · MoE 是"静态骨架+动态强度" · 魔方不是严格立方体

"整网" = 两件正交的事

宏观物理网络 ≠ 微观算子网络

宏观 · 立方体
全部卡（8192+），按视图重排、按状态上色。无算子细节。
微观 · 代表单元
一个选中单元（1 副本 / 1 EP 域）画全算子 + 时间，旁边标 ×N 广播。

不傻画 8192 份算子。整网感靠"一份 × N 副本"的乘法关系传达。在宏观里选一个单元 → 驱动微观详情，这就是宏观↔微观的桥。

MoE：别把它画错

专家不是立方体坐标

专家住哪张卡 = 静态（部署定死）→ 是卡上的一个标注，不是子立方体（专家是 HBM 里一块权重，不是电路）。

哪些专家被选中、传多少 token = 动态（routing 决定）→ 进泳道（expert-GEMM 段）+ 辉光。

卡间 A2A：能跟谁通信是静态（EP 域内固定骨架），通信多大是动态（只调线宽 / 辉光）。

魔方不是严格立方体（塌缩轴）

已由代码验证抓到

并行维不是互相正交的 —— EP 会「折进」别的轴

把并行配置想成魔方时，容易默认它是个干净的立方体：边长相乘 = 卡数。但会算错。 落地写 deployment.ts 时，运行时验证立刻抓到：tp8 × dp1024 × ep4 × tp8 = 32768 ≠ 8192。

原因：EP 不是独立的空间轴——推理时它折在 TP 域内（节点内路由）、训练时折在 DP 轴内。硬把它当一条边，就把同一批卡重复计数了。

spatial 空间轴
tp · pp · dp：三者乘积 = N，可给独立空间维度。
collapsed 塌缩轴
ep：折入 DP(训练)/TP(推理)，不再乘进 N。
virtual 虚轴
sp：与 TP 同域，无独立体量。

修法：给每个维度标 kind（spatial / collapsed / virtual），保证空间轴乘积 = N，让代码诚实反映"不是严格立方体"，而不是假装成干净魔方。这也印证了方案里的漏洞 #14——早验证早暴露，留到画立方体重排才发现返工会大得多。

监控优先 · 已定 首要任务是 B 运维监控。并行如何映射硬件不是为了教学，是为了把监控看清楚——8192+ 卡的红绿平铺读不懂，按并行维重排后异常会显出"形状"：TP 组整块红 = 该 Host 拖累 AllReduce；DP 副本整块红 = 数据管道；PP 竖板慢 = bubble；EP 域几张热 = 专家倾斜。形状直接对应根因类别——这就是"看清楚"。
