> 基于 2026-07 设计讨论与原型迭代（v3 → v18）整理。配套原型：`index.html`（=v18）。
> 上游文档：《大模型 AI Infra 训练监控可视化设计文档》（十二章）、v2 组件原型（十个组件）。

## 1. 定位与总体原则

产品定位：从"拓扑说明书"走向"性能诊断仪"。工程师在**逻辑整网**与**物理集群**之间需要无缝的缩放语义——宏观发现"哪台机器病了"，微观下钻"它为什么病了"。

交互设计的四条根本原则：

1. **一个任务意图驱动一组视图状态。** 透镜、形态、排列、时间轴都是"能力"，用户带着的是"问题"（哪里红了/为什么慢/越界去哪了）。好的联动不是控件间互相通知，而是任务入口一键编排全部状态（场景剧本，见第 8 章）。
2. **物理是唯一位置基准（Anchor）。** "逻辑是虚的，硬件是会烧的"——所有告警最终落到会烧的链路、OOM 的 IP。逻辑结构永远以透镜（着色）、连线、下钻内容的形式叠加在物理画布上，绝不与物理争夺位置通道。
3. **同一批实体，多种呈现。** 8192 颗芯片是唯一的"真身"，物理机柜、逻辑魔方、层级堆叠、五种魔方排列都是它们的重新摆位（Morphing）。实体身份（颜色、筛选、焦点、选中态）跨所有呈现保持连续——这是"双生视界"的实现基础。
4. **视觉通道分工，绝不同屏全画。** 位置=物理/魔方/堆叠（互斥形态）；着色=策略/健康/任务（互斥透镜）；连线=通信（随时间阶段过滤）；下钻=侧板内容（不占画布）；时间=第四维（全局时间轴）。

## 2. 画布分区：IDE 式三分屏

```
┌─────────────────────────────────────────────────────────────┐
│ 顶栏：透镜(互斥) · 场景开关 · 模式B入口 · 全局标量曲线 · 告警条 │
│ 场景剧本栏：巡检 / MoE诊断 / 找慢副本 / 单机深潜 / 部署审视 / 重置 │
├──────────────┬──────────────────────────────────────────────┤
│ 左：2D 索引区  │ 右：统一 3D 拓扑（单场景单相机）                 │
│  页签1 层级剖面 │   形态：物理层级 / 逻辑魔方(×5排列) / 层级堆叠    │
│  页签2 整网映射 │   Probe(下钻侧板) · HUD(总线遥测) · viewlab      │
├──────────────┴──────────────────────────────────────────────┤
│ 模式 B 面板（泳道 Trace / busbw 基线）——打开时挤压上方，非悬浮遮盖 │
├─────────────────────────────────────────────────────────────┤
│ 底部：播放控制 + 训练步阶段轨（TP→PP→EP→DP）+ 播放头            │
└─────────────────────────────────────────────────────────────┘
```

分区职责：**左侧=目录/索引**（抽象、全量、可点选），**右侧=世界本身**（空间、实体、相机飞行），**底部=时间**，**模式 B 面板=显微镜**（按需展开）。左右不重复：层级堆叠的认知价值由左侧承担，右侧堆叠形态是它的 3D 镜像而非替代。

模式 B 采用 IDE 式分屏而非悬浮抽屉：打开时压缩上方两栏（3D 视口重算、左侧连线重新锚定），保证三个区域同时可见、可交叉对照。

## 3. 全局状态机：单一状态源

所有视图共享一份状态 `S`，任何交互只改状态，渲染层各自响应。这是联动不乱的根基。

| 状态 | 含义 | 谁改它 | 谁响应 |
|---|---|---|---|
| `lens` | 着色透镜（健康/TP/PP/DP/EP/SP/CP/任务） | 顶栏按钮、剧本 | 左侧三画布、右侧全部实体、魔方刀面、viewlab 语义说明 |
| `level` | 当前层级 L7→L0 | 左侧层级行、下钻自动跳转、剧本 | 相机飞行缩放、层级边界高亮、机柜/Host/芯片分级淡化 |
| `focusHost / focusChip` | 焦点实体 | 左右任意点击 | 焦点框（跨三形态 Morph）、Probe、泳道对象、通信线锚点 |
| `filterHost + rel` | Smartscape 下钻 + 关系维度 | 点击实体、关系按钮、魔方点选 | 全视图非关联压暗、L2 折叠为 8 大芯片、计数徽章、归属链连线 |
| `selLayer / selCard` | 整网映射选择（模型侧） | 整网页签矩阵 | 3D 承载卡高亮（魔方=水平切片）、显存构成面板、矩阵裁切高亮 |
| `focusPod` | 所选超节点（SN0-63） | L5/L6 实体块 | 相机飞至该 Pod、3D 体块高亮、归属连线与计数跟随 |
| `form + cubeMode` | 形态 + 魔方排列 | 形态/排列按钮、剧本 | 8192 芯片位置权重混合 Morph、附属物（轴标/刀面/标尺）显隐 |
| `phase / t / playing` | 训练步时间 | 时间轴、播放、泳道点击、剧本 | 左侧锚定弧线、3D 通信线（物理+魔方内）、矩阵前向/反向扫动、泳道播放头、HUD 主导链路、标量曲线 |
| `anomaly` | 异常注入 | 场景按钮、剧本 | 全视图红标、告警条（可点击=定位剧本）、泳道加时块、busbw 回落指纹 |

约束：着色互斥（一次只回答一个"谁和谁一组"）；连线按当前阶段过滤（不画全量拓扑）；算子/显存永远下钻不平铺。

## 4. 层级模型与相机语义（L7→L0）

| 层级 | 对象 × 数量 | 左侧呈现 | 右侧相机 |
|---|---|---|---|
| L7 Global | 全球调度域 ×1 | 缩略行 | 全场远景 |
| L6 SuperCluster | 超节点 ×64 | 64 块（前 8 块标池成员色，全部可点） | 场地俯视 / 飞至所选 SN |
| L5 Service Pool | 服务池（组）×1，含 SN0-7 | 8 块（SN0·本=L4），可点 | 池行视角，偏向所选 SN |
| L4 Pod | 本超节点（=SN0）×1 | **128 计算柜逐柜 × 每柜 8 槽细条 + 32 互联柜**，可悬停点击 | 框住本 Pod |
| L3 Host | 计算节点 ×1024（32×32） | 逐格画布 | 飞至焦点机柜 |
| L2 Chip | 950DT ×8192（256×32）；下钻时折叠为焦点 Host 的 8 大芯片 | 逐格/折叠画布 | 贴近焦点 Host |
| L1 Die / L0 Core-Group | 结构占位 | 占位行 | 钻至焦点芯片（爆炸细节） |

关键设计：

- **点目录=相机飞行。** 左侧选层即单相机连续飞行缩放，这要求实体空间上真实包含（L2 装在 L3 里、L3 装在 L4 里）——分层带状排布做不到这一点（v3 的教训）。
- **命名去歧义。** L5 是"组"，成员是超节点 SN0-7，其中 SN0 就是 L4 的 Pod——成员块标注 `SN0 ·本(=L4)`，避免"L5 显示 Pod 和 L4 重叠"的困惑。
- **遮挡治理（zoom-in 三件套）。** 下钻时：① 机柜分级透明（L3=0.13，L2 以下=0.06）；② 其余 Host/芯片材质整体淡化（L3=0.22 / 更深=0.10），焦点 Host 与其 8 颗芯片改用**独立不透明网格**保持清晰；③ 焦点机柜保留蓝色线框作空间参照。
- **层级切换必须有非相机反馈。** 堆叠形态选中层平台描边变亮、标签变白、其余压暗；物理形态选 L5 池框发亮、选 L4 底座发亮——防止相机位移不明显时用户以为"没切换"。

## 5. 选择与下钻交互（Smartscape 式）

参考 Dynatrace Smartscape：**点击实体 → 关联集合 → 各层折叠**。

- **入口**：左侧 L3/L4 画布格、右侧 3D 方块，点击即下钻（再点同一实体取消）。
- **关系维度**：TP组（本机 8×NPU）/ PP链（同副本 16 段）/ DP环（同 Stage 64 副本）/ EP对端（专家互联）。默认"跟随阶段"随时间轴自动切换；**魔方内点选时自动继承当前透镜维度**（点芯片=选中它所在的透镜组）。
- **各层响应**：L2 折叠成焦点 Host 的 8 颗大芯片（其余 8184 颗标注"已折叠"）；L3/L4 非关联压暗、关联描边；层标题变计数（`关联 17/1024`、`关联 3/128 柜`）；右侧 3D 同步压暗。
- **连线三组，端点全部锚定真实单元**：① 纵向下钻线 Host→8 芯片；② 关系对端弧线（带箭头，超过 14 条汇总为"+N 已描边"）；③ 归属链 柜N→SN0→SuperCluster（端点带标签）。文字标注一律放在层间空隙，不压在密集网格上。
- **两套选择系统**（当前并存，见第 11 章统一计划）：层级页签的"实体下钻"（filterHost）与整网页签的"层/卡选择"（selLayer/selCard）。后者点层→3D 亮起承载卡（魔方=一整个水平切片），点卡→反查层段+显存构成+TP 分片说明+参数组绑定公式。

## 6. 魔方体系：一批立方体的五种堆法

核心洞察（用户直觉的确认）：**TP/PP/EP/DP 不构成严格正交立方体**——TP/PP/DP 正交可共存于标准魔方，EP 是稀疏子集，DP 兼有"复制"性质。因此魔方不是一个视图，而是"同一批 8192 颗立方体 + 多种排列"：

| 排列 | 堆法 | 回答的问题 |
|---|---|---|
| 标准 | X=TP×8 · Y=PP×16(=模型深度 48 层) · Z=DP×64 | 逻辑耦合距离；配合透镜刀面看"切哪个维度" |
| TP 切片 | 8 面权重墙沿 X 拉开，一面墙=全 Pod 同槽位切片 | 同槽位系统性故障（芯片批次、槽位散热） |
| PP 流水 | 16 段横向展开，左 Stage0 → 右 Stage15 | 哪段流水线慢 / 气泡在哪 |
| DP 平铺 | 64 副本 8×8 摊开各自成板 | 找慢副本 |
| EP 聚簇 | 64 专家芯片抽出成 8×8 专家墙（= MoGE 组结构），其余压致密块 | 路由倾斜、越界专家分布 |

配套机制：排列切换有 0.8s 重排飞行动画（实体身份连续）；标准排列下有轴标、透镜刀面（TP 竖切/PP 横切/DP 纵切）、Y 轴层段标尺（S0·L1-4…）、选层紫色切片；**魔方内通信线**随时间轴在晶格里呼吸（TP=行内环 / PP=列接力 / EP=星状 / DP=纵向环，异常时红线飞出晶格指向 SN1）；每个透镜附一句语义说明并提示对应专属排列。

魔方在整体架构中的定位：**不是第五种结构，而是策略图层的导航镜像**——物理按策略染色时，魔方按同轴切片同步染色，Morphing 本身就是"逻辑地址⇄物理地址"的双向映射可视化（折叠后可验证"TP 线仍在柜内、EP 线跨柜拉长"的物理层级铁律）。

### 6.1 选型论证：为什么这几种排列最适合表达 P、切分与层级

**论证一：形式与数据结构同构。** 可视化的最优形式是与数据内在代数结构同构的形式。TP×PP×DP 是三个正交索引的笛卡尔积，三维晶格是它唯一无损的空间嵌入——任何一维展开（列表）或二维投影（热力图）都必须牺牲至少一个维度的邻接性，而"邻接=通信耦合"正是我们要表达的核心语义（X 相邻=同节点 UB 高频，Y 相邻=流水线接力，Z 相邻=副本同步）。同理，L7→L0 层级是包含偏序，空间包含+相机缩放是它的同构形式：树图/缩进列表能表达从属，但丢掉物理邻接与规模感，而"这 8 颗芯片装在这台 Host 里、这台 Host 装在这个机柜里"恰恰是告警定位需要的那部分信息。

**论证二：诊断任务决定投影方式（感知精度层级）。** 依据 Cleveland-McGill 图形感知实验的结论，人对"共线位置/长度"的比较精度远高于"颜色/面积/深度"。由此每个诊断问题各自决定了一种排列：找慢副本是 64 个个体的量级比较，在标准魔方里 DP 是深度轴——透视缩短加互相遮挡使比较失效，必须摊平成并列小板做共线比较（DP 平铺=small multiples 的 3D 特例）；找慢段是 16 段的有序比较且自带方向语义（数据自左向右流），横向展开与流水线心智模型对齐（PP 流水）；同槽位系统性故障是"按 TP 序号聚类的离群检测"，只有把同槽位芯片重排为空间邻接（8 面墙）才能让离群一眼浮现（TP 切片）。**排列即预计算**：把用户本来要在脑中完成的"按某维度分组对比"预先用空间位置做掉。

**论证三：稀疏子集必须抽取重排。** EP 不是第四个正交轴——64 个专家散布在 1024 台 Host 上的稀疏子集，在任何正交排列里都只是噪声散点，负载对比无从谈起。对稀疏子集唯一有效的手段是"抽取-重组"（extract & regroup）：把专家从晶格中抽出堆成致密墙。MoGE 的组结构（8 组×8 专家）恰好给出墙的天然二维索引——v19 注入 Pangu Pro MoE 真实数据后，8×8 专家墙就是论文 Figure 1 的分组路由结构本身。

**论证四：备选形式的否决记录。** 邻接矩阵——通信量合适（已用于模式 B 通信矩阵页签），但作为主视图丢失物理位置这一唯一告警落点；Treemap——层级占比合适，丢失邻接与形状恒常性；平行坐标——多维趋势合适，8192 实体过密且无空间语义；多屏并列小魔方——违反"同一批实体"原则，用户须在屏间自行建立对应关系，而 Morphing 让系统替用户保持对象恒常性（object constancy）。

**不变量约束。** 五种排列共享同一实体身份：颜色（透镜）、选中（筛选/焦点）、下钻状态跨排列连续，0.8s 重排动画维持对象恒常性——因此"换排列"对用户来说是"换问题"，而不是"换数据"。这是多排列方案成立的前提，也是它优于"多个独立视图"的根本原因。

## 7. 时间维与模式 B（漏斗式工作流）

**时间轴**：一个训练步 = TP 前向 AllReduce → PP 接力 → EP AllToAll → DP 梯度同步。播放/拖拽驱动：左侧锚定弧线切换、3D 通信线（物理场景与魔方晶格内同步）、整网矩阵前向逐段扫过 + DP 阶段反向扫过（梯度回传，重算层闪绿=在线重算）、参与实体脉冲、HUD 主导链路切换、顶栏 Loss/LR/MFU 推进。

**模式 B 深潜面板**（对标 Nsight/Perfetto 的入口级呈现）：

- **泳道 Trace**：对象=当前下钻/焦点 Host 的 8×NPU（TP0-7 双轨 Compute/Comm），随左侧下钻联动；播放头与底部时间轴同源，已执行块亮/未执行块暗，点击泳道可拖时间；传统 Top-K vs MoGE 均衡对比（热点切片拖长 → 红色 Bubble vs 刀切齐 → 全掩盖）；画布按设备像素比渲染保证清晰。
- **busbw 基线**：busbw–消息大小曲线（512B→16GB 对数轴），灰带=历史基线 ±8%；异常时呈现"峰值后回落"的拥塞指纹并自动标注。
- **触发式抓拍闭环**：注入异常 → 告警条出现"已自动抓拍 Step N → 点击回放" → 点击执行完整定位剧本（见下章），实现文档的"巡航→抓拍→诊断"漏斗。

## 12. 软硬结合：状态 → 视觉形态映射（v16 起落地）

核心原则：**软件态是"预期"，硬件态是"测量"，诊断=两者的偏差。** 软件做结构层（分组、底色、时间刻度、连线的"该有哪些"），硬件做测量层（明度、形变、流速、告警），偏差自动高亮（泳道 Bubble 即原型）。

同一像素上的软硬结合：**色相=软件分组，明度=硬件利用率**；**形变**留给容量类硬件态（显存水位=Host 侧填充条高度）；故障=形态突变而非仅变色；**连线=软硬复合**（存在与颜色=软件通信域，粗细与流速=硬件带宽，线型=物理链路健康）。

选择器正交化：**着色镜头**（健康/策略/straggler/故障/任务，互斥）×**连线镜头**（自动·随阶段/通信域/互联流量/物理链路，独立单选）×**工况**（预训练/Prefill/Decode，规划中——重定义时间轴结构与全部基准值）。

| 硬件态 | 左侧剖面 | 3D 物理/魔方 | 泳道 | 顶栏/KPI |
|---|---|---|---|---|
| 利用率/MFU | 格子明度 | 方块明度 | Compute 块长度 + 计数器轨 | MFU 曲线 + 层级"典型/红/峰"摘要带 |
| 显存水位 | 悬停详情 | 方块侧填充条高度 | — | 红区占比（规划） |
| straggler | 描边闪烁 | DP 平铺"矮一截" | Bubble 红纹 | Top-N 分诊条 |
| 故障 | 红+标记 | 形态突变 | 缺块 | 故障计数 |
| 带宽/拥塞 | 弧线粗细 | 连线颜色/亮度=占用 | Comm 块 + UB 带宽轨 | COMM BUS |
| 链路健康 | — | 线型：绿常/橙降级/红中断带缺口 | — | — |
