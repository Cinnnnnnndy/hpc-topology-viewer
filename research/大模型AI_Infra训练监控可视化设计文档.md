> 整理自 2026-07 与 Gemini 的设计讨论，面向"大模型训练/推理运行时监控诊断工具"的产品与 UI/UX 设计。

## 1. 产品定位

从"拓扑说明书"走向"性能诊断仪（Performance Profiler）"。核心目标是帮助工程师在**逻辑整网**与**物理集群**之间建立无缝的缩放语义：既能宏观发现"哪台机器病了"，又能微观下钻"它为什么病了"。

目标用户分两类，决定了双模式架构（见第 5 章）：

- SRE / 值班算法研究员 —— 在线稳定性护航，需要低开销的实时大盘
- 算子 / 框架底层优化工程师 —— 离线极致调优，需要微秒级 Trace 回放

## 2. 核心训练对象的视觉标识体系

不同对象的生命周期和物理含义不同，用**色彩编码 + 形态差异**建立视觉锚点：

| 对象 | 通俗隐喻（面包厂） | 色彩 | 形态与呈现 |
|---|---|---|---|
| 权重 Weights / 偏置 Biases | 配方旋钮 / 默认底料 | 深蓝/靛青（稳重、体量大） | 矩阵块/热力图；节点大小=参数量；悬停显示 NaN/Inf 比例与分布直方图 |
| 激活值 Activations | 流水线上的半成品面团 | 明亮绿/青（流动、活跃） | 带方向的粗管道；管道粗细=显存占用（显存大头） |
| 残差 Residuals | 质量直通车 | 同激活值 | 跨层虚线/跳跃弧线（Skip Connection），梯度的"高速公路" |
| 梯度 Gradients | 经理的修改意见 | 紫/暗红（变化率、敏感度） | 权重旁的"影子矩阵"；闪烁边缘对比梯度爆炸/消失，标注 Clipping 阈值线 |
| 优化器状态 Optimizer States | 首席大厨的记事本 | 灰/暗调（后台常驻） | 权重底部的"基座/抽屉"，默认折叠；展开显示显存占用（通常为权重 2-4 倍） |
| Loss | 顾客差评扣分 | 橙/红（警戒线） | 全局标量，常驻顶部 Dashboard 折线图 |
| 学习率 LR | 听劝程度 | 金黄（引导线） | 全局折线图 |
| 动态缩放因子 Loss Scale | 放大镜（防微小梯度被抹平） | — | 突降（溢出）时在时间轴上闪烁红点标记 |

### 流式与关联关系的表达

- **前向流**：激活值自左向右流经各层，节点边缘顺向光效动画
- **后向流**：梯度反向流动；前向激活管道"变暗变细"（显存释放），紫色梯度管道激活
- **重算（Activation Checkpointing）**：常驻激活=实心绿块；重算激活=绿色虚线框，后向流经时瞬间闪实（在线重算）后熄灭
- **参数组绑定（Parameter Group）**：点击任一 Weight，自动高亮其对应 Gradient、Momentum m/v，用包围盒框在一起；悬停优化器状态弹出更新公式，符号与界面对象连线映射

## 3. 概念层级：算子 → 整网 → 模型 → 框架 → 物理集群

- **算子（Operator）**：最小原子动作（MatMul、Add、Softmax…），底层 CUDA/C++ 实现
- **整网（计算图）**：算子按依赖关系连成的"流水线图纸"，纯逻辑结构
- **模型**：整网结构 + 具体权重数值（静态）
- **框架（PyTorch / MindSpore / MindSpeed / vLLM）**：将逻辑映射到物理——编译整网、执行并行切分脚本、把算子分配到具体 GPU

一句话：**模型是灵魂，整网是骨架，算子是细胞，框架是外科医生，超节点是肉身**。并行策略（P）就是把巨大灵魂塞进有限肉身的"切割与缝合"技术。

## 4. 并行策略体系与物理映射

### 4.1 各策略的本质区别（切的维度不同）

| 策略 | 切什么 | 通信频率 | 硬件要求 | 一句话 |
|---|---|---|---|---|
| TP 张量并行 | 层内大矩阵 | 每次矩阵乘后（极高频，All-Reduce） | 必须机内 NVLink | 几张卡合力拧一个超沉的螺丝 |
| PP 流水线并行 | 层与层之间 | 阶段边界传递（低频） | 普通网络即可 | 车间接力，注意流水线气泡 Bubble |
| DP 数据并行 | 输入数据 batch | 仅末尾梯度同步（最低频） | 可跨机柜 | 分订单，各做各的最后对答案 |
| SP 序列并行 | 非核心层（LayerNorm/Dropout）的序列长度 | 进出特定层时 | 与 TP 绑定（TP-SP），机内 | 顺手撕草稿纸省显存 |
| CP 上下文并行 | 核心 Attention 层的超长文本 | Ring Attention 环形传递（中高频） | 可跨机，需较好网络 | 一本小说几人分段读、交换笔记 |
| EP 专家并行 | MoE 专家分布 | AllToAll（动态、浪涌式） | 专家多则必然跨节点，吃光纤带宽 | 术业有专攻，路由员派单 |
| FSDP/ZeRO | 权重本身切片 | All-Gather 借用/归还 | — | 配方撕成 N 份，用时互相借 |

### 4.2 叠加关系：正交且分层，不冲突

各策略切分**完全不同的维度**（数据量 / 模型深度 / 模型宽度 / 序列长度 / 专家），像长宽高一样正交叠加。每张卡获得多维坐标 `[DP_rank, PP_rank, TP_rank, (EP_rank)]`，即"魔方坐标系"。

### 4.3 物理层级铁律（按通信带宽需求排布）

1. **超节点内部（NVLink 级）**：TP + SP —— 最高频通信，必须机内
2. **机柜内跨节点（InfiniBand/RoCE 级）**：PP —— 中频接力，相邻 stage 放同机柜
3. **跨机柜全局网络**：DP —— 低频梯度同步，可散布机房各处
4. **EP 贯穿全局**：专家少则机内，200+ 专家必然跨节点，AllToAll 依赖光纤带宽——强监控优先级最高

## 5. 双模式监控架构（漏斗式工作流）

| 维度 | 模式 A：实时热力大盘 | 模式 B：时序泳道回放 |
|---|---|---|
| 隐喻 | 火灾报警器 / 心电图 | 黑匣子 / 显微镜 |
| 受众 | SRE、值班研究员 | 算子/框架优化工程师 |
| 时机 | 在线（训练进行中） | 离线（异常后 / 专项调优） |
| 指标 | MFU、温度/功耗、显存水位、吞吐 Tokens/s | 微秒级算子耗时、Overlap 比例、Bubble 大小 |
| 采集开销 | 极低（秒级轮询） | 极高（全量打点，只能短时开启） |
| 对标 | Grafana + Prometheus + DCGM | Nsight Systems / PyTorch Profiler (Perfetto) |

### 联动机制：异常触发快照（Triggered Dump）

1. **巡航**：热力大盘在线护航，某机柜群爆红（如 MoE 数据倾斜）
2. **抓拍**：系统自动在接下来 N 个 Step 开启深层 Profiling，记录 Trace 后立即关闭探针
3. **诊断**：工程师点击告警记录，无缝切入泳道回放，逐帧定位通信掩盖失败点

产品架构"一横一纵"：横向广度是 Dashboard（全局大盘、拓扑魔方、热力图），纵向深度是 Profiler（框选时间段/GPU → 右键"分析此段 Trace" → 下钻泳道图）。

## 6. 时序泳道图（Trace Timeline）设计

- **双轨设计**：每张卡至少拆两条子轨——Compute Stream（MatMul、Top-K Gating）与 Comm Stream（AllReduce、AllToAll）
- **Overlap 视觉语义**：
  - 健康：通信块完全落在计算块"阴影"内 → 绿色/低对比度
  - 瓶颈：通信块突出计算块（计算轨空闲 = Bubble）→ 高亮红色条纹，这是拖慢 MFU 的罪魁祸首
- **MoE 通信浪涌**：MoE 层通信轨色块瞬间密集，用热力渐变表示带宽占用率
- **Profiling 开销**：采样监控或特定 Step 打点，避免探针本身拖慢集群

### 案例：传统 Top-K MoE vs 盘古 MoGE（arXiv:2505.21411）

- 传统 Top-K：自由路由 → 热点专家挤爆某卡 → 各卡 Compute 长度参差 → AllToAll 被迫等待 → 满屏红色 Bubble；拓扑图上连线粗细不一，热点卡标红
- 盘古 MoGE：专家分 8 组对应 8 卡，每 Token 每组强制选 1 → 各卡负载物理强制均衡 → Compute 块刀切一样齐 → AllToAll 100% 被掩盖，一片健康绿；拓扑连线粗细完全对称

## 7. "魔方"逻辑拓扑视图

### 7.1 坐标定义（默认排列逻辑）

- X 轴 = TP（最内层连续维度，耦合最紧）
- Y 轴 = PP（数据沿 Y 轴层层流动）
- Z 轴 = DP（每个 Z 切面 = 一个完整模型副本）

逻辑魔方展示**计算依赖关系**（离得越近耦合越高）；物理机柜视图展示该数学结构被拆解塞进真实网线后的状态。

### 7.2 降维排查交互（切片 Slicing）

1. 全局俯视（DP 视图）→ 发现某副本慢，点击进入
2. 切面透视（PP 视图）→ 看哪段流水线 Bubble 过长
3. 微观下钻（TP/EP 视图）→ 展开超节点 8 卡，看 NVLink 带宽或 MoE 路由是否堵在某卡

### 7.3 动态挑战

静态训练下魔方形状固定（TP=8, PP=4 写死在脚本里）；MoE 推理/动态路由下，数据流走向随时间变化（某阶段 8-16 卡被占用，与专家数/数据量相关；200+ 专家、Top-K 取 4/8）——需要时间维度的呈现（见第 5 章双模式）。

## 8. 多结构融合：以物理为基准的"透镜"体系

**核心问题**：物理结构、卡间通信线、算子整网、并行策略四类结构可否同屏？

**答案：绝对不能全部同屏**（会成"视觉毛线球"）。工业界最佳实践：

1. **基准（Anchor）= 物理集群**。"逻辑是虚的，硬件是会烧的"——告警最终定位的是烧掉的 NVLink、OOM 的 IP、拥堵的交换机。画布底层永远是机柜/超节点/GPU 的空间排列
2. **透镜（Lenses）按需叠加**（高德地图隐喻）：
   - 并行策略透镜 = 行政区划涂色（GPU 按 TP 组 / PP 阶段染色）
   - 通信透镜 = 实时路况（红黄绿通信线，按需浮现）
   - 算子下钻 = 点击某 GPU 弹出详情面板（当前执行的 MatMul / AllReduce）
3. **万卡规模的宏观降噪**：聚合渲染（机柜级/节点级汇总），下钻才展开单卡

## 9. 魔方 × 物理透镜的融合：双生视界（Digital Twin）

三种递进的融合方案：

- **方案 1 分屏联动（Brushing & Linking）**：左侧逻辑魔方，右侧物理机房；悬停魔方某 TP 切片，物理视图中散落各机柜的对应 GPU 同色亮起。价值：一秒看出"逻辑上紧耦合的一组张量计算物理上竟跨了两个机架"
- **方案 2 魔方降维为透镜控制器**：物理机架是主视图，魔方缩为左下角 3D 导航雷达/过滤器；在魔方上"切一刀"（选 DP 面）= 对物理视图应用 DP 透镜
- **方案 3 时空折叠动画（Morphing）**：魔方解体 → 小方块按真实物理接线重组成超节点/机架，内部连线随之展开（适合首页/汇报演示）

### 联合诊断案例

物理大盘 Rack_5 温度带宽全红 → 点击 Rack_5，联动魔方亮起零散方块 → 发现 PP_Stage_3 与 PP_Stage_0 挤在同一机柜抢光纤 → 结论：物理拓扑与逻辑拓扑错位，调度配置错误。

## 10. 未决问题与后续方向

- **数据源**：解析标准 Trace（PyTorch Profiler .json / CANN Profiling）vs 自研底层采集组件
- **渲染性能**：万卡规模需 WebGL / 3D 游戏引擎方案（Three.js、deck.gl），DOM 渲染撑不住
- **默认粒度**：面向算子工程师（微观）还是算法人员（宏观收敛+利用率）决定默认视图
- **Profiling 开销控制**：采样率、触发式快照的 Step 窗口大小

## 11. 通信可视化深化：借鉴 NCCL 生态（对照昇腾 950 超节点 / HCCL）

### 11.1 NCCL 与昇腾 950 超节点的关系

二者不在同一层，是"软件通信库 vs 硬件系统"的关系，真正的对位是 **NCCL ↔ HCCL**：

| NVIDIA 生态 | 华为昇腾生态 | 层级 |
|---|---|---|
| NCCL（集合通信库） | HCCL（支持 AllReduce/Broadcast/AllGather/ReduceScatter/AllToAll，Ring/Mesh/递归 HD 算法） | 软件：通信原语 + 算法 + 拓扑感知 |
| NVLink / NVSwitch（机内） | 灵衢 UnifiedBus 柜内铜缆 FullMesh | Scale-up 互联 |
| NVL72/NVL144 超节点域 | Atlas 950 SuperPoD：8192 张昇腾 950DT，互联总带宽 16PB/s，单跳时延 2μs→200ns，跨柜往返 7μs→3μs | 超节点 |
| InfiniBand / RoCE（Scale-out） | 灵衢 2.0 柜间 OCS 全光交换（跨柜带宽 >1TB/s）+ 集群组网 | 跨节点/跨柜 |
| nccl-tests | hccl_test / ascend-dmi | 基准测试 |
| NCCL fabricId 拓扑层级（DC 内 / DC 间 / 断连） | 超节点内 / 超节点间 / 集群间层级 | 拓扑感知抽象 |

结论：监控工具面向 950 超节点时，被监控对象是 HCCL 的通信域、算法与链路，但 NCCL 公开资料中的**方法论（拓扑感知、Channel、算法选择、Profiler 事件模型、基准判读）可整体平移**。HCCL 在超节点上还有自己的强项可作为监控指标：通信算子重编排使通信量降低 25%、直发 NPU Vector 核使小包耗时降 90%、整网通信耗时降 50%——这类"优化前后对比"本身就适合可视化呈现。

### 11.2 四篇文章各自可借鉴的显示思路

**① NCCL 跨 DC 通信与拓扑感知（2025.07）**

- 核心机制：fabricId 把链路分为三级（DC 内 / DC 间 / 断连）；Ring/Tree 构建时**先在域内成环/成树，再用松散端跨域拼接**，保证跨域链路数恰为 `2*(nDC-1)`；`NCCL_SCATTER_XDC=1` 散射让每个环从不同节点跨域，避免单设备瓶颈（无散射时 4 GPU×400Gbps 会要求单跨域链路扛 1.6Tbps）
- 可视化借鉴：**分层带宽域视图**（超节点内 / 柜间 / 集群间三级底色）+ **环/树叠加在物理拓扑上**，高亮"环在哪里跨越慢链路"；散射开关前后对比动画；跨域链路计数与理论值校验

**② NCCL 2.27 弹性与可观测（2025.07）**

- Communicator Shrink：训练中动态剔除故障卡、重建通信器（默认模式=计划内重构，错误模式=中止恢复）→ 可视化为**弹性事件时间线**：故障卡标记 → 通信器收缩动画 → 拓扑重建
- SHARP 在网计算：AG/RS 卸载到 NVLink/IB 交换机，GPU 通信 SM 占用 16→6 → 泳道图应区分"通信消耗 GPU SM"与"交换机在网卸载"两种形态，SM 占用本身是监控指标
- Profiler 增强（直接指导我们的数据源设计）：ProxyOp/ProxyStep 事件、GPU 原生全局时钟打点、`recordEventState` 把实时网络拥塞提示注入时间线、通信器元数据（名称/ID/节点数/通道数按实际用量上报）

**③ NCCL 2.12 PXN 与 AllToAll 翻倍（2022.02）**

- 轨道优化拓扑（rail-optimized）：所有服务器的 NIC-k 接同一台叶交换机 L-k；PXN 路径 = GPU 先经 NVLink 挪到与目的地同轨的 GPU，再走本地 NIC，**避免穿越第二层脊交换机**（3 hop → 1 hop），并支持 8 消息聚合提升消息速率
- 可视化借鉴：**消息路径 hop-by-hop 视图**——同一对 GPU 通信，对比"跨轨 L0→Spine→L3 三跳"与"PXN 单跳"；轨道按颜色分组；AllToAll 的 O(N²) 消息矩阵热力图

**④ nccl-tests 实测判读（Asterfusion 案例）**

- 核心视图：**busbw–消息大小曲线**（512B~16GB 对数横轴）。健康曲线单调爬升后在大消息区饱和成平台；案例中 256MB 峰值 48.23GB/s 后**回落**至 33.94GB/s，即 ECMP 对大象流哈希不均导致交换机间链路拥塞的典型指纹
- 判读要点：看峰值与平台区而非平均值；数据量要压到带宽上限；达标判据 = 跨交换机 busbw ≈ 网卡直连（案例优化后 ~195GB/s）
- Channel 解析方法论：从 NCCL_DEBUG=INFO 日志提取每条 Channel 的环路 rank 序列、每 hop 传输类型（P2P/CUMEM vs NET/GDRDMA）与所用 NIC，聚合出"机间流量落在哪些 NIC 对"——直接定位轨道错接/ECMP 问题。这就是我们工具"通信透镜 + 算子下钻"的日志级数据源
- busbw 换算（nccl-tests 官方）：busbw = algbw × 系数，AllReduce 为 2(n-1)/n，AllGather/ReduceScatter/AllToAll 为 (n-1)/n

### 11.3 落地：监控工具新增的三类通信视图

1. **集合通信构建视图**：在分层物理拓扑上动画展示 Ring/Tree 的构建与跨域拼接，散射开关对比，慢链路穿越点计数与告警
2. **消息路径诊断视图**：选定源/目的 GPU，展示实际 hop 序列（NVLink→NIC→Leaf→Spine），对比 PXN/非 PXN，标注每跳带宽与拥塞
3. **通信健康度基线视图**：busbw–size 曲线常态化采集（类 nccl-tests/hccl_test 定期跑），叠加历史基线带，"峰值后回落""平台低于直连基线"自动告警

## 12. 最终结合方案：统一驾驶舱（一张画布，四个维度）

对"多结构能否同屏"的最终回答：**能结合，但结合的方式不是把四张图叠在一起，而是认识到四种结构是同一批 GPU 对象的四个属性维度**，各自占用不同的视觉通道，天然不冲突：

| 结构 | 视觉通道 | 图层规则 |
|---|---|---|
| 物理集群 | **位置**（唯一基准画布） | 永不切换、永不隐藏——所有告警的最终落点 |
| 并行策略 | **着色/分组** | 互斥图层（颜色通道只有一个）：TP/PP/DP/EP 单选 |
| 卡间通信 | **连线** | 可叠加图层，且是时间的函数——随训练阶段（TP AllReduce→PP 接力→EP AllToAll→DP 同步）自动变化 |
| 算子整网 | **下钻内容** | 不占画布：点击 GPU 弹出侧板（当前算子构成、参与的通信组、承载的层切片） |
| （时间） | **第四维** | 时间轴 Scrubber/播放控制连线与热点，统一驾驶舱与回放模式共用 |

魔方在此方案中的定位：**不是第五种结构，而是策略图层的导航镜像**——物理机架按策略染色时，魔方按同轴切片同步染色，让工程师在"逻辑正交世界"和"物理真实世界"之间保持双向对照（即第 9 章方案 2 的落地形态）。

视觉降噪三原则：着色互斥（一次只回答一个"谁和谁一组"）；连线按阶段过滤（只画当前时刻活跃的通信，不画全量拓扑）；算子永远下钻不平铺。

对应 v2 原型第九组件"统一驾驶舱"。其实现直接复用前两个组件的成果：**位置维度 = 第七组件的 3D 等距魔方 + 时空折叠 Morphing + 硬件遥测 HUD**（逻辑魔方 ⇄ 物理机柜一键折叠，折叠后可验证"TP 线仍在柜内、EP 线跨柜拉长"的物理层级铁律）；**通信维度 = 第八组件的 NCCL/HCCL 语义**（域内短链 / 接力链 / 跨柜星状 AllToAll 附跨柜链路计数与散射建议 / 全局同步环，COMM BUS 遥测条随阶段自适应切换主导链路）。

## 附：v2 升级原型相对 v1 的增量

v1（Gemini 原版）实现了：并行策略切分探究器（TP/PP/DP/SP）、泳道诊断仪（传统 MoE vs MoGE）、3×3×3 魔方-机架联动。

v2 补齐讨论中提出但 v1 未实现的能力：

1. 并行策略探究器增加 **EP（专家并行）** 与 CP 独立说明
2. **实时热力大盘模式**：GPU 网格呼吸式热力刷新、MoE 浪涌告警（爆红+告警横幅）
3. **时间轴回放模式**：播放/暂停/拖拽 Scrubber，逐帧查看 MoE AllToAll 星状路由连线与热点漂移
4. **透镜系统**：以物理机架为基准，叠加 TP/PP/DP/EP 着色透镜与通信线透镜；点击 GPU 下钻算子详情面板
5. 魔方-机架联动增加 **悬停联动（Brushing & Linking）** 与 EP 动态路由高亮
6. **3D 并行魔方映射探究仪**（复刻对话中的交互组件）：4×4×4=64 卡逻辑立方，◀▶ 轮播 TP/PP/DP/EP 透视维度；硬件通信总线 HUD（NVLink 900GB/s vs InfiniBand 50GB/s vs PCIe 32GB/s 按模式高亮）；PHYSICAL RACK MAPPING STATUS 面板（8 机柜×8 槽位亮块显示通信组分布）；"切换至物理机柜视图"开关触发时空折叠 Morphing 动画；悬停方块读取硬件遥测探针
