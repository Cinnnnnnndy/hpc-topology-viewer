> 目标读者：本项目后续开发者。本文把讨论中形成的完整想法收敛成一份可执行的改进计划，
> 并附一份**漏洞与风险清单**（这是本文最重要的部分之一）。

## 0. 一句话概括

**首要任务 = B 运维监控**（§13 #7 已定）。把「模型算了什么（算子）→ 框架怎么切（TP/PP/EP/DP）→ 摆到哪台机器（放置）→ 跑得好不好（时间/通信掩盖）」这条链可视化，**目的是让运维在 8192+ 卡里把异常看清楚**——并行映射是监控的可读性工具，不是为教学。做成**一个可重排的立方体阵 + 若干动态叠加层**：

- **一张卡 = 一个单位立方体**；并行配置是一个**超立方体（魔方）**，物理机房只是它的一种投影。
- **切视图 = 换一种堆法**（按 TP / PP / DP / EP / 物理 / 算子重新分组）；核心价值是**让异常按并行维聚类、显出形状**（见 §9.1），形状对应根因类别。
- **时间和状态永远只改颜色/辉光，不改位置**（见 §2 铁律）。
- **状态面接真实遥测上色**；**泳道图**承载动态：计算/通信随时间的排布、bubble、以及通信是否被计算「掩盖」。

## 1. 背景：现在有什么

| 资产 | 现状 | 复用价值 |
|---|---|---|
| `parallelMap()` (data.ts) | TP/SP/EP/PP/DP 的**唯一真值源**：`groupOf`(组)、`peersOf`(通信对端)、度数、`collectiveOf` | 极高——它已经是逻辑坐标+通信域的真值 |

| `StatusView` (运行状态) | 2D 仪表：4 镜头(热力/流量/通信域/物理链路) + L7→L0 层级轴 + KPI | 中——收编为「算子镜头在各级的投影」 |
| `STEP_DECOMP` | 每工况的 计算/通信/访存 **占比** | 中——给掩盖比例，但**不是调度**(见漏洞#4) |

**关键判断**：融合的骨架大半已就位。真正缺的是三样**新数据模型**（见 §6）和一个**统一的空间抽象**（立方体重排）。

## 2. 核心抽象：三个平面 + 一条铁律

把所有内容强制归到三个平面，每个平面**只用一种编码通道**，不许串台：

| 平面 | 内容 | 编码通道 | 静/动 |
|---|---|---|---|
| **结构面** | rank↔坐标↔位置、专家→卡、层→stage、切片→tp rank | **空间位置 + 分组**（= 立方体的堆法） | **静态** |
| **流动面** | 每时刻算什么/通信什么、bubble、掩盖 | **时间（泳道 X 轴）+ 上下轨** | **动态** |
| **状态面** | t 时刻的 util/掉队/故障/流量/专家热度 | **颜色/辉光**（叠在结构面上，由流动面游标驱动） | **动态** |

> **铁律：动态事实只能改变「颜色/辉光/游标位置」，永远不能改变「卡的空间位置」。**
> 位置这个通道被结构面独占。一旦时间也去动位置，位置通道就被抢，立方体立刻读不懂。

## 3. 五级翻译链（模型 → 机器）

框架是「模型 → 机器」的翻译器，这是把整条链讲清楚的主线：

```
① 模型算子图     embedding → 48×[Attn, MoE] → norm → LMHead        「算了什么」
      │ 框架读并行配置 (MindSpeed/Megatron parallel_state)
② 并行拆解       TP8·PP5·EP2·DP·SP·VPP —— 每个算子被切/复制         「怎么分」
      │ 框架建 device mesh + ProcessGroup
③ 逻辑 rank 网格  多维 mesh，每格 = 1 rank，组内 = 1 个集合通信域     「谁和谁一组」
      │ placement 策略把 rank 摆到卡 (rank → NPU)
④ 物理放置       global rank r → Pod/Host/NPU 坐标                  「跑在哪台机器」
      │
⑤ 物理 3D 阵列   全部卡 + 卡间通信链                                ← 右侧全景
```

`parallelMap()` 现在**同时压缩了 ②③④**。改进要把它们**显式拆开、可视化**——让用户看到「框架把一个算子切成 8 份 TP、摆到这台 host 的 8 张卡上」。

## 4. 各并行策略如何把算子拆进物理体系

每种 P 是对**算子图**的一条「重写规则」，并**注入一类集合通信算子**——正是这些通信算子变成 3D 里的卡间连线：

| 策略 | 对算子图做什么 | 注入的通信算子（= 3D 连线） | 落在物理哪一级 |
|---|---|---|---|
| **TP** | 单个 matmul 沿 hidden 切 N 片，每卡算一片 | 每层 2× **AllReduce** | L3 host 内 8 卡（UB 高带宽） |
| **SP** | LayerNorm/Dropout 沿 sequence 切 | **AllGather + ReduceScatter**（与 TP 同域） | 同 TP 域 |
| **PP** | 48 层按 stage 切 5 段，每段一组卡 | stage 间 **P2P send/recv** | L4 Pod 内跨 host |
| **VPP** | 每 stage 再交错成多个 virtual chunk（减 bubble） | 更密的 P2P（交错调度） | 同 PP（**只体现在时间轴**） |
| **EP** | 64 专家分到 EP 组各卡，token 按 router 路由 | **All-to-All**（dispatch + combine） | 推理 L3 host 内 / 训练折进 DP |
| **DP** | 整模型复制 N 份，各喂不同 batch | 梯度 **Ring-AllReduce** | L5/L6 跨 Pod |

## 5. 立方体多视图重排（空间主干）

### 5.1 为什么魔方是对的

一套并行部署 = 一个 **N 维整数点阵**。每张卡有坐标 `(pp, dp, ep, tp[, sp, cp, vpp])`，各维边长 = 各并行度。
**逻辑超立方体 → 物理 3D 是一次「展平/投影」，这次展平正是框架的 placement 策略**。
所以「不同视图 = 不同堆法」= 选哪几个 mesh 轴映射到屏幕 X/Y/Z，其余轴用**颜色/嵌套/时间**编码。

### 5.2 能堆出的视图（同一批卡重排）

| 视图 | 空间轴 | 你会看到 |
|---|---|---|
| **物理**（现有） | pod→cabinet→blade→slot | 真实机房，TP 组紧、DP 副本远 |
| **TP** | TP 组 8 卡成块，沿 DP 平铺、沿 PP 堆 | 张量切片如何贴在 host 内 |
| **PP** | X = stage 0→4（5 竖板），板内 DP×TP 填面 | 流水线 + bubble（需配泳道） |
| **DP** | 每副本 = 一个相同子立方体，网格平铺 | 「整网 = 一份 × N 副本」直接可见 |
| **EP** | 按 EP 域聚簇，域内摆专家 | MoE 专家分布 + A2A |
| **算子/部署** | 按 (第几层, 哪个算子) 摆位 | 逻辑模型排布 |

### 5.3 实现机制：换嵌套顺序

物理布局本就是**嵌套投影**（slot⊂blade⊂cabinet⊂pod）。切视图 = 换嵌套分组函数（如 `tp⊂ep⊂dp⊂pp`）。
`FullPodScene` 用 InstancedMesh，位置本就由 `k` 算出；`parallelMap.groupOf(k,dim)` 给逻辑坐标。
**把位置从「物理公式」改成「视图布局函数 `layout(k, view)→(x,y,z)`」即可**，切换时每帧 lerp 实例位置。

### 5.4 视图切换的诊断价值

**同一条集合通信连线，在不同视图里长度天差地别。** EP 视图 A2A 短（专家挨着），切回物理视图**同一批连线被拉长、散布整个机房**——切换动画直接呈现「这个逻辑紧凑的 EP 组物理上跨了半个 Pod，通信要走多远」，用于诊断「通信为何慢」这类问题。

## 6. 宏观/微观两层模型（解决"代表单元 vs 全阵列"的矛盾）

> 这是把前面所有想法拼起来的**关键澄清**，避免自相矛盾。

**"整网"其实是两个正交的东西，之前被混为一谈：**
- **整个物理网络** = 全部卡（8192+）→ 由**立方体（宏观）**承载。
- **整个算子网络** = 完整算子 DAG → 只在**一个代表单元（微观）**上画全。

| 层 | 画什么 | 画多少 | 承载 |
|---|---|---|---|
| **宏观** | 全部卡，按视图重排，按状态上色 | 全阵列，**无算子细节** | 立方体 3D |
| **微观** | 一个选中单元（1 副本 / 1 EP 域 / 1 host）的完整算子 + 时间 | **代表单元 + ×N 广播标注** | 算子图 + 泳道 |

**桥梁**：在宏观里选一个单元 → 驱动微观详情。整网感靠 DP 视图「一份 × N 副本」的乘法关系 + 广播倍率徽标传达，**不靠逐份渲染算子**。

## 7. MoE 专项（最值得画，也最容易画错）

MoE = **静态骨架 + 动态强度**的叠加，两者分属不同平面：

- **专家住在哪张卡 = 静态**（部署时定死）→ 结构面，卡上的**子卡标注**（**不是子立方体**，见漏洞#1）。
- **哪些专家被 topk 选中、收多少 token = 动态**（数据相关、逐 step 变）→ 流动面（expert-GEMM 块）+ 状态面（辉光）。
- **卡间 A2A = 静态拓扑 + 动态强度**：**能跟谁通信是静态的**（EP 域内，`peersOf(k,'ep')`）；**通信多大是动态的**（routing）。
  → 3D 里画**固定的 A2A 骨架** + 游标推进时只调**边宽/辉光**。

**topk = 每 token 的扇出上界**（activatedExperts=8 → 每 token 最多 8 路 A2A），聚合成**卡×卡流量矩阵**，
被 EP 域大小和 topk 框死 → 可画的稀疏矩阵。MoGE（分组 Top-1、IS=0）把这个动态矩阵约束成**均衡、结构化**的。

## 8. 泳道图与「掩盖」（流动面）

### 8.1 每级一套泳道语义

| 在哪级看 | 泳道（行） | 道上放什么 | 「掩盖/overlap」表达 |
|---|---|---|---|
| L4 Pod | PP 的 5 个 stage | 每 microbatch 的 F/B 块 + stage 间 P2P | **bubble 空档** + VPP 交错掩盖 |
| L3 Host | TP 的 8 张卡 | 各卡算子分片 + AllReduce | 计算下叠 AllReduce，**重叠=通算掩盖成功** |
| L2 Chip | 单卡算子时序 | load/compute/comm/store 相位 | 访存与计算重叠 |
| L0 Core | Cube/Vector/CPU 三种单元 | 微 kernel | 单元间流水 |

### 8.2 掩盖的画法

每条泳道**上下双轨**：上轨 = 计算算子，下轨 = 通信算子，时间轴对齐。
- 通信条被计算条完全罩住 → **已掩盖**（overlap OK）。
- 通信条露出计算条之外 → **暴露的通信开销**（红）——正是 `STEP_DECOMP` 的「通信%」来源，也是优化靶子。

泳道与 3D 靠**同一时间游标**联动：拖游标 → 泳道竖线扫过 → 3D 对应时刻的算子点亮、集合通信连线发光。

## 9. 监控与问题定位（首要任务，见 §13 #7）

### 9.1 立方体在监控下的重定位

首要任务是 B 运维监控，"并行如何映射硬件"服务于监控——目的是**看清楚**。这条决定了立方体/并行映射的地位：它不是叙事主角，而是**让监控可读的工具**。

**核心价值：并行重排让异常显出"形状"，形状直接对应根因类别。** 8192+ 卡的红/黄/绿平铺是读不懂的；按并行角色重排后：

| 异常在立方体里的形状 | 指向的根因类别 |
|---|---|
| 某个 TP 组（一台 host 8 卡）整体变红 | 该 host 硬件/散热/单卡拖累整组 AllReduce |
| 某个 DP 副本整块变红 | 该副本的数据分片/落盘/输入管道问题 |
| 某个 PP stage 一条竖板慢 | 该 stage 负载不均 / bubble 异常 |
| 某个 EP 域内几张卡热 | 专家负载倾斜（routing 不均 / 热专家） |
| 散点无规律的红 | 偶发抖动 / 个别卡故障 |

**所以立方体的每个视图 = 一种"异常按哪个并行维聚类"的透镜。** 运维切视图 = 换一个维度问「异常是不是沿这条轴成组的」，成组即命中根因类别。这是"看清楚"的精确含义。

### 9.2 问题定位路径

**问题全都活在时间里，不在空间里**（掉队卡、热专家、暴露通信、故障）。监控的权重落在**时间轴 + 状态辉光 + 游标拖动 + 异常标记**。定位路径（DAVIS 根因链已是种子，需延伸到算子/通信级）：

> 泳道异常（通信条露出 = 掩盖失败 / bubble 异常大 / 某专家条超长）→ 点它 → 立方体点亮肇事卡+链 → 切视图看异常形状 → 下钻看哪个算子/专家 → 根因链。

## 10. 数据模型（新建，且必须从 SSOT 派生）

| 对象 | 职责 | 从谁派生 / 来源 |
|---|---|---|
| `ingest`（监控数据摄入契约） | 定义外部喂入的 schema：作业并行配置、`rank→节点`放置、逐 rank 时序遥测（util/通信字节/掉队标志/时间戳） | **外部集群/作业**（监控首要任务下的新增地基） |
| `deployment` | device mesh 形状 + `rank↔物理坐标` placement + **算子实例↔rank 双向映射** | 优先取自 `ingest`；缺失时按 `parallelMap` 推导并标注"推导值" |
| `opGraph` | **参数化**的 transformer 算子 DAG（1 层子图 × 层数 + 每种 P 的重写规则），每算子带 kind；**职责是给热卡贴标签** | `WORKLOAD` + `deployment`（结构）；数值接遥测 |
| `layout(k, view)` | 各视图的卡位置布局族（物理/TP/PP/DP/EP/算子） | `parallelMap.groupOf` |
| `schedule` | 泳道用的**事件调度**（op 起止 + comm 起止 + 掩盖判定） | 早期 `STEP_DECOMP` + microbatch 数（新参数）合成；后期接算子级 profiling trace |

**架构纪律**：`opGraph` / `layout` / `schedule` **必须派生自** `parallelMap` / `deployment`，绝不独立造第二套真值，否则必然发散。**监控数据（`ingest`）是真值来源，与派生真值必须在 `deployment` 处对齐**——冲突时以摄入的真实放置为准。

## 12. ⚠️ 漏洞与风险清单

> 以下逐条列出**尚未解决、或容易做成"看着对其实骗人"的点**，按严重度排序。

### 严重（会导致误导或返工）

1. **专家不是立方体坐标。** 专家比卡多（如 16/卡），它是**卡的一段时间工作**，不是物理子结构（die/core 是电路，专家是 HBM 里一块权重 + 一段计算）。**绝不能给专家一个 3D 位置**，只能在泳道上表现为 expert-GEMM 段。这条最容易做错。

2. **8192+ 卡视图切换的动画性能。** 每帧 lerp 上万实例位置 + 重排 peer mesh 连线，A6（15488 卡）压力更大。必须 LOD / animate-then-settle / 连线在动画中降级。相机 framing 也要随视图变。

3. **推理 H²P 的 mesh 会「随子相位重构」。** 推理时同一批 8 卡：注意力相是 DP2×TP4、路由专家相是 TP2×EP4、共享相是 TP8——**同一硬件在一次前向的不同子相位里扮演不同 mesh 角色**。所以推理的「TP 视图」和「EP 视图」其实是**不同时间切片**，不是纯空间重排 → 视图与时间在推理里**部分耦合**。训练则干净（TP8·EP2·PP5 整步不变）。文档/UI 必须区分训练 vs 推理两种语义。

4. **`STEP_DECOMP` 只有占比，不是调度——泳道画不出来。** 掩盖/bubble 需要**事件级起止时间**（op A [t0,t0+d0]、comm B [t1,t1+d1]，重叠判定），占比生成不了。P2 需要一个 `schedule` 生成器 + **microbatch 数**这个当前缺失的参数。这块工作量被低估了。

### 中等（会削弱解释力或一致性）

5. **DP 视图的「物理散布」故事在单 Pod 硬件下偏弱。** 代码现在把 DP 副本**平铺在一个 Pod 内**（"逻辑上跨 Pod"）。所以「逻辑紧凑 vs 物理散布」的震撼动画对 EP/TP 强、对 DP 弱，除非建模多 Pod。需诚实标注。

6. **A2A 只能画聚合流量，不是 token 轨迹。** 画不出「token#47 去哪」。UI 必须写清「线宽 = 该步窗口聚合流量」，否则是假精确。

7. **专家数 / EP 度 / 卡数的一致性红线。** 当前 `WORKLOAD` 是 **64 专家**（8 激活/4 共享），用户设想的「200+ 专家」是更大模型（Ultra MoE / DeepSeek 档）。**载入哪个模型就只能讲哪个的数**，且「专家/卡」必须**推导**（`routedExperts/EP`，且 TP 还会再复制——组合 TP×EP 下的专家放置比"总数/EP"更微妙），绝不写死。

8. **交互复杂度预算。** 现已有：8 级轴 + 视图切换器 + 4 镜头 + 方向过滤 + 指标 + 时间游标 + 框架选择 + 并行度配置 + 泳道。**认知过载风险高**。必须定「主控件 vs 渐进披露」，否则界面失控。

9. **一张卡同一时刻常常同时算+通信**（overlap 是目的）。「这张卡现在在干嘛」是个**集合**不是单值。3D 点亮当前算子时要允许一张卡同时亮计算+通信两个通道。

### 已澄清 / 已升级

10. **✅ 已解决：用户与任务已定为 B 运维监控**（§13 #7）。合成数据仅作早期 bootstrap，最终接真实遥测（见 #11）。

11. **⬆️ 已升级：监控数据分两类，工作量差一个量级。** ① **监控遥测**（逐 rank util/通信字节/掉队标志/时间戳）——轻量、可行，从一开始就接真实。② **算子级 profiling trace**（逐算子起止，8192 rank × 数千算子）——数量级更重，先合成/推导、后期再接。二者不可混为一谈；泳道的原理版可先靠合成 `schedule`，但状态面的监控上色必须接真实①。

12. **SSOT 会变多。** 加了 `ingest`/`deployment`/`opGraph`/`layout`/`schedule` 后，要保证派生真值全部源自 `parallelMap`/`deployment`，且与摄入真值在 `deployment` 处对齐。需要架构约束 + 测试守住。

### 补充点（原方案外）

13. **可测不变量：每个视图必须是同一张卡集合的一个置换。** `layout(k,view)` 对固定 view 必须是**双射**（每张卡在每个视图里恰好出现一次、不重叠、不丢失）。这是**可单元测试**的硬约束——3D 可视化难测，但这条能测，应作为 P1 的验收门槛。

14. **collapsed/virtual 轴要显式标注。** 推理 TP4+EP4 挤在一个 host（塌缩轴）、VPP/SP 无独立空间（虚轴）——这些在立方体里给不了正交空间维度，必须用「同域嵌套 + 标注」或「只进时间轴」表达，不能假装它们是独立的立方体面。

15. **⬆️ 已升级为安全关键：「近似铺满」在监控下会误导排障。** `nearestDivisor` 让 PP5/EP2 除不尽 8192 → 锯齿状超立方体 + padding。教学下可接受，但监控下若显示的 rank↔卡映射是"近似铺满"而非真实放置，会把运维引向错误的卡。**监控模式必须优先用摄入的真实放置；用不到真值而退回推导时，必须显著标注"推导/近似，非真实 rank 图"**，且如实画填充/空格。

### 监控优先带来的新增漏洞

16. **数据摄入契约是新地基（原方案没有）。** 监控让 app 从"自包含合成"变成"数据驱动前端"。需要定义 schema：作业并行配置、`rank→节点`放置、逐 rank 时序遥测。这是 P0 新增的主工作项，不是小改。

17. **实时 vs 快照 / 时间语义。** 现有 replay（step 0–60 合成）要换成**真实时间窗口**。需明确：拉取周期、数据时延、"现在"指哪一刻、数据陈旧如何提示。

18. **告警 / 阈值语义。** 监控要回答"什么算异常"，不只是原样上色。现有 `loadState`（绿/黄/红）阈值是拍的，需要基线/阈值/同比机制，否则"异常标记"无依据。

19. **实时数据规模。** 8192+ rank × 多指标 × 时间轴的采集/传输/存储/查询/降采样，都是新工程；前端要能承受持续刷新而不卡（与漏洞#2 的渲染性能叠加）。

## 14. 收敛后的整体结构（一句话）

**立方体（结构面）只管静态的"部署成什么样"，泳道（流动面）管动态的"随时间怎么跑"，辉光（状态面）是叠在立方体上、被泳道游标驱动的动态皮肤；三者共用一个选区 + 一个时间游标；动态永远不动位置。**
MoE 的复杂性被劈成「静态放置 + 静态可通信拓扑」（进立方体）和「动态激活 + 动态流量」（进泳道/辉光），不再纠缠。
"整网"分成「整个物理网络（宏观立方体）」和「整个算子网络（微观代表单元 + ×N 广播）」两件正交的事。
